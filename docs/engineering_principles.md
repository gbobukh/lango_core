# Принципы разработки (Lango Core)

Краткий канон инженерных ориентиров для этого репозитория. Детали модулей и потоков см. в `project_map.md` и остальных документах в `/docs`.

## Приоритет качества и темпа

Для этого проекта приоритет разработки фиксируется явно:

- сначала **правильно, аккуратно, системно, как положено**;
- скорость важна, но **не ценой архитектурных долгов**;
- временные hot-fix решения допускаются только как исключение при авариях и должны сопровождаться последующим системным исправлением;
- изменения в UI/runner-логике должны быть совместимы с текущей архитектурой (полиморфные step-type механики, существующие контракты и валидации).

## Поддерживаемость и явные полиморфные контракты

**Неочевидный полиморфизм недопустим.** Если набор значимых полей или структура JSON зависит от дискриминатора (например, `step_type` у шага сценария), такое поведение не может опираться только на неформальное знание или разрозненный код без единой точки истины.

Требования:

- для каждого значения дискриминатора зафиксирован **отдельный проверяемый контракт**: какие поля обязательны, какие не используются, какова допустимая форма вложенных данных (JSON и т.п.);
- контракт **выражен в коде** в явном, узко ограниченном месте (не размазан по формам и админке без общего слоя правил);
- при сохранении и при необходимости в рантайме выполняется **валидация по этому контракту** с понятными сообщениями об ошибках;
- в `/docs` дан краткий обзор смысла каждого варианта и **ссылка на модуль/слой контракта в коде**, без попытки держать полную схему только «в голове» или только в прозе документа вместо проверяемых правил.

Иными словами: разный смысл у тех же полей или у одного JSON-поля допустим **только вместе** с явным, проверяемым контрактом в коде и сопутствующей документацией-обзором; отсутствие такого контракта считается дефектом проектирования.

**Реализация для шага сценария:** контракт при сохранении из админки собран в модуле `service_builder/scenario_step_contracts.py` (функция `validate_scenario_step_cleaned`). Форма `ScenarioStepForm` вызывает его из `clean()` — это единая точка правил по полям и по форме `action_config` для каждого `step_type`. Для `API_CALL` поля `action_type` и `action_config` в сохраняемых данных не используются: они принудительно обнуляются при валидации (не нужно заполнять и не должно оставаться старого JSON в БД после сохранения шага как API Call).
